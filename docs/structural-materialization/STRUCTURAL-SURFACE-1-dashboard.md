# Structural Surface Materialization 1
## Dashboard

### Concept A: Priority Rail + Action Stream
### 1. Structural hierarchy diagram
```text
[Top: Day Readiness + Next Action]
[Primary Column: Upcoming / Needs Attention stream]
[Secondary Column: Availability + Event Type readiness snapshots]
[Context Layer: Inline trust notes attached to affected cards]
[Deep Layer: Expandable diagnostics drawer (closed by default)]
```
### 2. Primary/secondary/contextual zones
- Primary: next commitments, immediate actions.
- Secondary: readiness snapshots.
- Contextual: local trust messages on affected rows/cards.
### 3. Progressive disclosure map
- Default: coordination only.
- Level 1: card detail.
- Level 2: issue reason.
- Level 3: advanced diagnostics drawer.
### 4. Expansion/collapse strategy
- Row expansion for meeting detail.
- One expanded trust detail at a time.
### 5. Responsive/mobile collapse behavior
- Collapse to: next commitment -> urgent action -> compact readiness chips.
- Secondary rail becomes stacked collapsible groups.
### 6. Trust visibility placement
- Trust chip near each impacted commitment.
- One lightweight section-level warning maximum.
### 7. Escalation placement
- Inline warning -> section alert -> modal only if unsafe mutation risk.
### 8. Interaction zoning
- Zone 1: quick actions.
- Zone 2: browse/filter.
- Zone 3: deeper investigation.
### 9. Density-risk analysis
- Risk: too many summary cards create dashboard wall.
- Control: cap visible summary modules; move extras to collapsible section.
### 10. Orchestration leakage analysis
- Leakage risk: status jargon in hero.
- Prevention: hero language only in user impact terms.

### Concept B: Timeline-First Coordination
### 1. Structural hierarchy diagram
```text
[Header: Today / Week toggle + confidence summary]
[Main: Chronological booking timeline]
[Side/Below: Action queue grouped by urgency]
[Context: Trust markers on timeline items]
[Hidden: Technical detail panel]
```
### 2. Primary/secondary/contextual zones
- Primary: timeline and action queue.
- Secondary: readiness summaries.
- Contextual: trust markers where timeline items are impacted.
### 3. Progressive disclosure map
- Timeline item -> detail panel -> root-cause note -> advanced data.
### 4. Expansion/collapse strategy
- Expand one timeline cluster at a time.
### 5. Responsive/mobile collapse behavior
- Timeline switches to agenda cards.
- Action queue becomes top priority strip.
### 6. Trust visibility placement
- Marker and one-line impact in each affected card.
### 7. Escalation placement
- Only block action when booking reliability is unsafe.
### 8. Interaction zoning
- Navigation/filter zone, coordination zone, recovery zone.
### 9. Density-risk analysis
- Risk: timeline plus queue overload.
- Control: show top 3 actions; defer rest.
### 10. Orchestration leakage analysis
- Leakage risk: displaying provider/system state transitions as primary timeline labels.
- Prevention: use human-readable booking state language.

### Validation Questions
- Does this still feel like scheduling coordination, not monitoring?
- Is scheduling intent more visible than system mechanics?
- Is trust contextual and non-dominant?
- Is orchestration complexity abstracted by default?
- Does it preserve calmness and tier continuity?
- Would non-technical users feel guided instead of monitored?
