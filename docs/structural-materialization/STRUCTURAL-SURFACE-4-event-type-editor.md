# Structural Surface Materialization 4
## Event Type Editor

### Concept A: Guided Stage Editor
### 1. Structural hierarchy diagram
```text
[Top: Event identity + publish readiness]
[Primary stages: Basics -> Scheduling Rules -> Participant Experience -> Safeguards]
[Context: In-stage trust/readiness checks]
[Hidden: Advanced behavior tuning]
```
### 2. Primary/secondary/contextual zones
- Primary: stage content and core decisions.
- Secondary: optional enhancements.
- Contextual: validation and readiness cues.
### 3. Progressive disclosure map
- Core fields -> conditional options -> advanced behavior panel.
### 4. Expansion/collapse strategy
- One stage open focus; completed stages collapse.
### 5. Responsive/mobile collapse behavior
- Stepper becomes vertical stage list; one active stage visible.
### 6. Trust visibility placement
- Section readiness chip per stage + overall publish confidence.
### 7. Escalation placement
- Blocking escalation only at publish when unresolved critical issues exist.
### 8. Interaction zoning
- Build zone, preview zone, validation/recovery zone.
### 9. Density-risk analysis
- Risk: settings-wall fatigue.
- Control: stage chunking and strict advanced collapse.
### 10. Orchestration leakage analysis
- Risk: backend rule engine language appears in field labels.
- Control: booking-outcome wording only.

### Concept B: Split Intent + Rules Canvas
### 1. Structural hierarchy diagram
```text
[Left: Event intent + participant promise]
[Right: Rule clusters and constraints]
[Bottom/Drawer: Publish checks and issue remediation]
```
### 2. Primary/secondary/contextual zones
- Primary: intent and key rules.
- Secondary: optional controls.
- Contextual: issue list tied to sections.
### 3. Progressive disclosure map
- Rule summary -> advanced rule refinements.
### 4. Expansion/collapse strategy
- Expandable rule clusters with “advanced refinements” per cluster.
### 5. Responsive/mobile collapse behavior
- Left/right becomes sequential: intent first, then rules, then readiness.
### 6. Trust visibility placement
- Trust only near affected rule groups.
### 7. Escalation placement
- Destructive rule changes require confirmation.
### 8. Interaction zoning
- Intent zone, configuration zone, confidence zone.
### 9. Density-risk analysis
- Risk: split view complexity.
- Control: collapse secondary metadata and keep one rule cluster active.
### 10. Orchestration leakage analysis
- Risk: exposing internal precedence resolution as user task.
- Control: system precedence remains implicit unless requested.

### Validation Questions
- Is this booking-design-first rather than settings-console-first?
- Is publish confidence clear and calm?
- Are advanced controls progressive?
- Does the structure reduce cognitive load on mobile?
- Does it preserve one product identity across tiers?
